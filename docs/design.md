## 1. 目的とスコープ

請求業務そのものは既存SaaSを継続利用する前提で、「請求の結果（売上）＋工数」を集計し、次の指標を月次で自動可視化する分析ツール。請求書発行システムではない。

可視化する主要4指標（構想メモ準拠）:

1. クライアント別の時間単価（売上 ÷ 投下工数）
2. スタッフ別の時間単価（帰属売上 ÷ 稼働工数）
3. スタッフ別の月次売上（帰属売上）
4. スタッフ別の売上目標達成率（帰属売上 ÷ 目標）

現状のデモ版はFIX版ではない。本番までに新規実装する機能が残っており、デモにある画面・機能がそのまま最終形ではない。デモ未実装の本番実装スコープは第15章に集約する。

## 2. 全体アーキテクチャ

3層構成。サーバ側ロジックは持たず、静的フロント＋GAS＋スプレッドシートで完結する軽量構成。

- フロントエンド: HLSレンタルサーバ上の静的ファイル（HTML/CSS/JS）。ビルド工程なし。
- バックエンドAPI: Google Apps Script（`gas/Code.gs`）をウェブアプリとしてデプロイ。
- データストア: Googleスプレッドシート（7シート）。

通信方式は **JSONP**（`backend.js` が `<script>` タグを動的生成してGASを呼ぶ）。静的サイトからCORS制約なしで呼べる反面、後述のセキュリティ制約がある。

```
[ブラウザ: index/staff/dashboard]
        │  JSONP (?action=...&token=...&payload=...&callback=...)
        ▼
[GAS Web App: doc/Code.gs  doGet → route_]
        │  SpreadsheetApp
        ▼
[Google Spreadsheet: worklogs ほか7シート]
```

### データフロー
- 参照系: `bootstrap`（マスタ＋工数）, `dashboard`（集計用フルデータ）。
- 更新系: `saveEntry` / `saveEntries` / `deleteEntry` / `upsertMaster` / `removeMaster`。更新時はGAS側ダッシュボードキャッシュを無効化。

## 6. 配賦ロジック（現状：工数按分方式）

構想メモ4章の(2)工数按分方式を採用。月内で、顧客売上をその顧客に投下した工数比でスタッフへ配賦する。

擬似コード（dashboard.js buildMonthModel）:

```
for each customer:
    if (customer.hours == 0 || customer.revenue == 0): continue   // ★ここが論点
    for each staff who logged hours on this customer:
        staff.revenue += customer.revenue * (staff.hoursOnCustomer / customer.hours)
```

**前提条件**: 顧客に「工数」と「請求」の**両方**がある場合のみ配賦が発生する。

## 7. 既知の論点・制約

### 7-1. 配賦の非対称（最重要・改修対象）
全体KPIの「税抜売上」は billing_data を**全件合計**する一方、スタッフへの配賦は「工数と請求の両方がある顧客」に限られる。したがって**請求はあるが工数記録がゼロの顧問先**が存在する月では:

- その売上は総売上には算入されるが、どのスタッフの帰属売上にも乗らない（＝未配賦）。
- スタッフ帰属売上の合計 ＜ 全体の税抜売上。
- 全体達成率（税抜売上÷目標）＞ スタッフ別達成率の合計。
- 顧客平均時間単価は分子に工数ゼロ顧客の売上が入り分母には入らず、実態より高めに出る。
- その顧問先は顧客別表に「工数0h・時間単価¥0」で表示される。

社労士事務所では「顧問料は毎月請求するが当月は工数未記録」が普通に起こり得るため、本番データでは高確率で顕在化する。現デモデータでは該当ゼロのため未発生。

→ **改修方針（案2・第8章で設計確定）**: 工数の裏付けが無い売上（工数ゼロ顧客・工程未計上・マッピング漏れ）は **顧客担当者マスタへ等分フォールバック**配賦する。さらに紐づけを品目コード基準＋工程（Prepare/Review）に精緻化し、時間単価・生産性が歪まないよう売上を2系統（配賦売上／工数対応売上）に分ける。

### 7-2. 主担当の出どころ
現状の「主担当」は工数の最多入力者から逆算しているため、**工数ゼロの顧客では主担当を決定できない**。案2では **customer_staff_master（顧客担当者マスタ・複数担当可）** を新設し、表示・フォールバックともにこのマスタ由来とする（第8-8）。

### 7-3. しきい値ハードコード
顧客時間単価の色分け（rate>=10000:良 / >=7500:中 / 未満:低）、スタッフ達成率の色分け（>=1.0 / >=0.85）が dashboard.js にハードコード。所長の基準に合わせ調整余地あり。

### 7-4. セキュリティ
JSONP＋URLクエリにトークンを載せる方式で、`config.js` にトークンが平文。初回デモ用途では許容だが、本番は認証・権限・監査ログ・レート制限・編集履歴の設計が必要（DEPLOY.md 注記済み）。リポジトリでは `config.js` を .gitignore 対象とし、`config.sample.js` をプレースホルダとして管理する。

### 7-5. スタッフマスタの規模
seed-sample スクリプトは15名分を含むが、現行デモは reseed-six（6名）で運用。マスタ・工数・目標は6名で整合済み。

## 8. 案2：工程対応・顧客担当者フォールバック配賦（設計確定・未実装）

> 状態: 2026-06-23 に設計を確定（adachi 合意。受領資料 `J-Office_Insight_管理/【DM】請求品目一覧.xlsx` を反映）。実装は未着手。実装時にデモデータも作り直す（第8-13）。
> この章は設計の正本。実装完了後に第4〜7章（現状仕様）へ反映する。
> 一部は所長回答待ち（第8-14）。

### 8-1. 狙い

現状の配賦（第6章）は「顧客単位・総工数按分」で、業務区分を見ない。また請求はあるが工数ゼロの顧問先で未配賦が生じ、指標が歪む（第7-1）。案2では次を実現する。

- 売上↔工数の紐づけを **(請求月, 顧客, 業務コード)** の粒度に精緻化。
- すべての業務区分に **工程（Prepare / Review）の振分率**を持たせ、売上を工程比で分けてから工数按分。
- 工数の裏付けが無い役務売上は **顧客担当者マスタへ役割対応でフォールバック配賦**（役務売上は全額が必ず誰かに乗る）。
- 売上を性質で **役務売上 / 配賦対象外売上（立替・税）** に分類。
- 売上を集計用途で **配賦売上 / 工数対応売上** の2系統に分離し、金額系と per-hour 系で使い分け。

### 8-2. 売上↔工数の紐づけ（業務コード基準・恒等マッピング）

受領資料により、**請求側の「業務コード（品目コード）」がそのまま業務区分**であり、工程振分率・名称も同一表に載ることが判明した。したがって:

- 紐づけキーは **業務コード**。`billing_data.invoiceItemCode`（業務コード）＝ `task_master.code` ＝ `worklogs.taskCode` で**恒等マッチング**（別途のマッピング列は不要）。業務コードは外部「かつ・かいしゅう」で確定・運用されている正本。
- 紐づけは **(請求月→作業月変換, 顧客, 業務コード)**（作業月変換は第8-5）。
- マッチング漏れ（請求にあるが業務区分マスタに無い業務コード）の売上は、顧客担当者へフォールバック配賦し、**データ品質警告**（第8-11）。空白コード `000`/`057` 等が該当（所長回答待ち）。

### 8-3. 工程（Prepare / Review）モデル

- **すべての業務区分が工程振分率を持つ**（「工程なし」という分岐は設けない）。分割しない区分は **Prepare:100 / Review:0** としてマスタ登録する運用。
- 振分率は業務区分マスタ（実体は子＝工程マスタ）で管理。受領資料の例：労務相談=100/0、給与計算=70/30、年末調整=70/30、LAI=25/75 など。Prepare/Review に限定せず **N工程・任意振分率（役務は合計100%）に拡張可能**な構造（当面2工程運用）。
- 工程は工数側に **別カラム `phaseCode`** で保持（業務区分名に内包しない）。工程入力UIは、ratio>0 の工程が1つだけなら自動選択/非表示にしてよい。

### 8-4. 配賦区分（役務 / 対象外）と消費税

業務区分マスタに **`配賦区分`** を持つ。0/0（Prepare:0/Review:0）の品目は性質で切り分ける。さらに**消費税は「税」であり売上ではない**ため、税抜売上からも除外する（3区分）。

| 区分 | 例 | 税抜売上(総) | 役務売上 | スタッフ配賦 |
|---|---|---|---|---|
| **役務**（配賦対象） | 給与計算026・労務相談001・スポット手続 ほか | 含む | 含む | する（工程比・工数登録対象・Σ振分率=100） |
| **対象外**（立替・実費） | 諸費用060 | **含む** | 含まない | しない（工数登録の選択肢に出さない・per-hour に入れない） |
| **消費税**（税） | 消費税080 | **含まない** | 含まない | しない（税込算出にのみ使用） |

- 消費税(080)は請求CSV上、**非080行(税抜)合計の10%として独立行**で入る（実データ 2026-03 で全社一致を確認・端数1社¥1差）。売上ではなく税なので**税抜売上にも含めない**。税込 = 税抜売上 ＋ 消費税。
- 027 マイナンバー管理は所長回答待ち（第8-14）。スタッフ配賦しないなら対象外、担当者へ載せるなら役務として比率設定（一旦は対象外で仮置き）。
- 対象外売上（立替）は「捨てる」のではなく**名前を付けて別建て**にする（第8-7・第8-11）。

### 8-5. 請求月の決定と作業月オフセット（2つの別軸）

「ズレ」が2種類あり、**別物として両方持つ**。

**(ア) 振替日 → 請求月**（取込時に売上の帰属月を決める）
- 口座振替は翌月引落しのため、**請求月 = 振替日の前月（振替月 − 1・固定）**。例: 振替日 2026/4/22 → 請求月 2026-03。
- 1ファイル＝1請求月（振替日は単一）。取込時に自動付与し、**手修正可**。

**(イ) 作業月 → 請求月オフセット**（配賦時に対応工数の月を遡って探す・グローバル設定）
- 設定画面で **当月請求(0) / 翌月請求(+1) / 翌々月請求(+2)**。事務所一律（品目別は持たない）。
- マッチングは「**請求月 B の売上 ↔ 作業月 = B − オフセット の工数**」。
- 人事オフィスは工数も請求月と同月計上の想定のため **基本は (イ)=0**（工数データ確認後に確定）。
- 設定値は設定保持の仕組み（設定シート/専用マスタ）に1件保存。

### 8-6. 配賦アルゴリズム

```
オフセット = グローバル設定（0/1/2）
各 (請求月 B, 顧客, 業務コード I) について:
  R = その業務コードの税抜報酬額合計（消費税080は取込時に税として分離済み・配賦処理に入らない）
  T = code == I の業務区分（恒等マッチング）
  作業月 M = B − オフセット(イ)
  W = (作業月 M, 顧客, 業務区分 T) の工数エントリ群
  A(role) = 顧客担当者マスタ(顧客) の role 役割の担当者集合

  ■ T が無い（マッチング漏れ）:
      R を A(全役割) へ等分（フォールバック）＋警告

  ■ T.配賦区分 == 対象外（諸費用 等の立替・実費）:
      R は「配賦対象外売上」へ計上（スタッフ配賦なし・役務売上に含めない・税抜売上には含む）

  ■ T.配賦区分 == 役務:
      各工程 p（振分率 r_p, Σr_p=1。r_p=0 はスキップ）:
        枠 = R × r_p
        Wp = W のうち phaseCode == p
        Wp あり → 枠を Wp の工数比で按分（工数対応売上）
        Wp なし → 枠を「顧客の p 役割担当者」へ等分（フォールバック・役割対応）
        （p 役割担当者が未設定 → 未配賦として残し警告）
```

**売上2系統**（同じ配賦処理の結果を2通りに集計）:
- **配賦売上**（金額系：帰属売上・月次売上・達成率用）＝ 工数按分分 ＋ フォールバック分。合計 = **役務売上**。
- **工数対応売上**（per-hour 系：時間単価・生産性用）＝ 工数按分分のみ（**フォールバック分は除外**）。

**売上の階層**: 税抜売上(総) = 役務売上 ＋ 配賦対象外売上（立替）。Σスタッフ帰属売上 = 役務売上。税込 = 税抜売上(総) ＋ 消費税（080・税抜売上には含めない）。

### 8-6-1. 数値例

A社・当月（オフセット0=当月請求）。担当者: Prepare担当=P、Review担当=R。

| 業務コード | 金額 | 配賦区分 | 工程比 | 当月工数 |
|---|---|---|---|---|
| 026 給与計算 | ¥100,000 | 役務 | 70/30 | X:Prepare 4h, Z:Prepare 1h（Reviewなし） |
| 060 諸費用 | ¥20,000 | 対象外 | — | 工数なし |

- 給与計算 Prepare枠 ¥70,000 → X ¥56,000 / Z ¥14,000（工数対応）
- 給与計算 Review枠 ¥30,000 → Review工数ゼロ → **Review担当 R へ ¥30,000**（役割対応フォールバック）
- 諸費用 ¥20,000 → 対象外 → 誰にも載らない／役務売上に含めない

役務売上 = ¥100,000 = X56k + Z14k + R30k ✓（= Σ帰属売上）
総売上(税抜) = ¥120,000（役務 ¥100,000 ＋ 対象外 ¥20,000）
工数対応売上 = X56k, Z14k のみ（R のフォールバック分は除外）→ 時間単価 X = ¥14,000/h
→ R は工数ゼロでも帰属売上¥30,000・達成率に反映（時間単価には載らない）。

### 8-7. 指標の使い分け（売上系統マトリクス）

per-hour 系は **工数対応売上**、金額系は **配賦売上（=役務売上）** を用いる（全レベルで一貫）。配賦対象外（立替・税）は per-hour にも帰属売上にも入れず、総売上にのみ計上して別建て表示。

| 指標 | 使う売上 | フォールバック分 | 配賦対象外分 | 工数=0 の表示 |
|---|---|---|---|---|
| スタッフ 帰属売上（月次売上） | 配賦売上(役務) | **乗る** | 乗らない | — |
| スタッフ 達成率 | 配賦売上 ÷ 役務目標 | **押し上げる** | 乗らない | — |
| スタッフ 直接時間単価 | 工数対応売上 ÷ 直接工数 | 反映しない | 反映しない | 「—」 |
| スタッフ 総合生産性 | 工数対応売上 ÷ 総工数 | 反映しない | 反映しない | 「—」 |
| 顧客 売上 | 総売上（税抜・全額） | — | 含む（内訳表示） | — |
| 顧客 役務売上 | 役務売上 | 含む | 含まない | — |
| 顧客 時間単価 | 工数対応売上 ÷ 顧客工数 | 反映しない＋「未記録¥X」別表示 | 反映しない | 「—（工数未記録）」 |
| 全社 総売上(税抜) | 全額 | — | 含む | — |
| 全社 役務売上 | 役務のみ | 含む | 含まない | — |
| 全社 顧客平均時間単価 | 工数対応売上合計 ÷ 顧客直接工数合計 | 反映しない | 反映しない | — |
| 全社 達成率 | 役務売上 ÷ 役務目標 | 含む | 含まない | — |

判断確定（2026-06-23）:
- per-hour 系（時間単価・生産性）は**顧客個別も含め工数対応売上に統一**。顧客一覧で 売上列（総額）と「時間単価 × 工数」は一致しないため、差分（未記録分・対象外分）を別表示で補う。
- 「総合生産性」も**工数対応売上ベース**。
- **売上目標は「役務売上」基準**。達成率分母を役務売上に揃えることで Σスタッフ達成率 = 全社達成率 が一致する。

### 8-8. 工数なし売上の各指標への影響（要点・フォールバックと対象外の区別）

- **フォールバック分（役務だが工数ゼロ）**: 役務売上のまま担当者へ載る。帰属売上・達成率に**反映**。時間単価・生産性には**反映しない**。
- **配賦対象外分（立替・税）**: 誰にも載らず役務売上にも入らない。帰属売上・達成率・時間単価・生産性のいずれにも**反映しない**。総売上にのみ計上し別建て表示。
- **工数ゼロ月の顧客時間単価**は「—（工数未記録）」（¥0 でも ∞ でもない）。その売上は担当者フォールバックで全社役務売上・スタッフ達成率に反映。
- これにより第7-1 の不整合は解消する。「全社総売上 ≠ Σスタッフ帰属売上」が残るが、それは**配賦対象外（立替・税）ぶんの意図的な差**であり、別建て表示で説明できる（バグではない）。

### 15-3. 売上（請求）データのCSV取り込み

| billing_data | CSV列 | 備考 |
|---|---|---|
| customerCode | [6] 関与先コード | 例 0001 |
| customer | [7] 関与先名 | |
| invoiceItemCode | [8] 内訳 | ＝業務コード（恒等マッチング） |
