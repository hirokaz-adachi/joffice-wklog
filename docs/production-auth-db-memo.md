# 本格提供に向けた認証・DB方針メモ

作成: 2026-06-18 ／ HLS（adachi）
位置づけ: デモ（静的フロント＋GAS＋スプレッドシート）から本番提供へ移行する際の、認証とデータベースの方針たたき台。実装は未着手。確定仕様ではない。
関連: [design.md](./design.md)（第7章 既知の論点／第8章 案2）

---

## 1. 背景・前提

- スタッフが工数登録するアプリ（`staff.html`）は、本番では **社員番号＋パスワードでのログイン運用**を想定。
- 本番のデータ基盤は **MySQL** を採用する想定。
- デモ段階では認証は実装しない（現行の GAS＋スプレッドシートのまま）。

MySQL（＝実バックエンド）が入る前提なら、認証は枯れた標準パターンで実装でき、難易度は低い。論点は「認証の難しさ」ではなく「GAS＋シート → 実バックエンド＋MySQL の移行をいつ・どう行うか」。

---

## 2. 想定構成

- ホスティング: 本番は人事オフィスの既存HP環境（**X-Server / エックスサーバー**）への配置を想定（デモはHLS自社のお名前.com 上 `tools.h-linksystems.com` で稼働中・別環境）。X-Serverは一般に **PHP＋MySQL** 可・Node等の常駐は不可。よって素直な構成は **バックエンド=PHP / DB=MySQL**。
- フロント: 既存の HTML/CSS/JS を**ほぼ流用**。`backend.js` の通信を JSONP から**同一オリジンの `fetch()`（PHPエンドポイント）**へ差し替える。
- 同一オリジンになるため、httpOnly・Secure な Cookie セッションが素直に使える（JSONP由来の制約が解消）。
- 将来 VPS 等に移すなら Node/Express・Python でも同設計で可。

## 3. 認証設計（標準パターン）

- `users` テーブル: 社員番号（ログインID）、パスワードハッシュ、**ロール（admin/manager/staff）**、有効フラグ、初期パスワード変更フラグ、作成/更新日時。
- パスワードは **bcrypt 等でハッシュ化**して保存（PHPなら `password_hash()` / `password_verify()`。平文保存は厳禁）。
- ログインAPIで照合 → **セッション**を発行（httpOnly・Secure・SameSite な Cookie。またはJWT）。各APIでセッション検証。
- **認可（重要）**: 画面を隠すだけでなく、各データAPIでロール・所属に応じたチェックを行う（詳細は下記「ロールと可視範囲」）。
- 任意で強化: 初回ログイン時の強制パスワード変更、連続失敗時のロックアウト、操作監査ログ、パスワードリセット導線。

セキュリティ上、GAS時代の「JSONP＋URLトークン」「画面だけのガード」は本番では用いない。

### ロールと可視範囲（RBAC）

ユーザー種別は3つ。可視範囲は次を基本とする（★は要件定義で確定が必要な前提）。

| ロール | 主な権限・可視範囲 |
|---|---|
| スタッフ（staff） | 自分自身の情報のみ。自分の工数の入力・編集。自分の実績（帰属売上・時間単価・達成率など**金額系も含めて開示OK＝確定2026-06-22**）。他スタッフや全社ダッシュボードは不可。※**スタッフ向けダッシュボードを公開するか自体は未定。現状は非公開前提で進める。** |
| マネージャ（manager） | **全スタッフ（staffロール）の情報を閲覧**（スタッフ別の工数・実績）。**他のマネージャおよび管理者の情報は参照不可**（自分自身の情報は閲覧可）。**金額系（売上・単価・目標・達成率）も閲覧可＝確定2026-06-22。** |
| 管理者（admin） | すべて閲覧・編集。マスタ管理、請求・目標データ、全社KPI、全スタッフ。**複数名を設定可（確定2026-06-22）。** |

実装の要点:

- `users` に `role`（admin/manager/staff）列。**マネージャの可視範囲は全スタッフで確定（2026-06-22）**のため、チーム単位の絞り込み（`staff` の `manager_id`／部署列）は不要。閲覧対象は `role='staff'` のユーザーに限定し、他マネージャ・管理者は除外する。
- 認可はサーバ側API（PHP）で実施。ログインユーザーの role／所属に応じてSQLでデータをフィルタする（フロントの画面出し分けは補助で、本当のガードはAPI側）。
- ダッシュボードは role により表示軸を変える（マネージャ＝スタッフ軸／管理者＝全軸）。**スタッフ向けダッシュボードは現状非公開前提のため、当面はマネージャ・管理者のみに提供（スタッフ公開の可否は別途判断）。**

要件定義で詰める点:

- ~~マネージャの範囲：全スタッフ か 担当チームのみ か~~ → **全スタッフ（staffロール全員）で確定（2026-06-22）。他マネージャ・管理者は対象外。チーム／manager-staff 紐付けは不要。**
- ~~マネージャに金額（売上・単価・目標）を見せるか~~ → **見せる（金額系も閲覧可）で確定（2026-06-22）。**
- ~~スタッフ本人に金額系（帰属売上・時間単価・達成率）を見せるか~~ → **見せる（本人の金額系は開示OK）で確定（2026-06-22）。ただしスタッフ向けダッシュボード画面の公開可否は未定（現状は非公開で進める）。**
- ~~管理者を複数置くか~~ → **複数置ける前提で確定（2026-06-22）。**

## 4. DB移行（シート → MySQL）

現行スプレッドシートの各シートはほぼそのままテーブル化できる。

| 現シート | テーブル（案） |
|---|---|
| worklogs | worklogs |
| staff_master | staff |
| customer_master | customers |
| task_master | task_types |
| billing_data | billings |
| staff_target_master | staff_targets |
| （新規） | users（認証） |

- 集計・配賦（時間単価、達成率、工数按分、案2の主担当フォールバック）は、アプリ層のクエリ／ビューで実装。シート版のロジックを移植する。
- 工数の所有者を表すため、worklogs と users（社員番号）を紐付け、認可チェックに使う。
- 権限管理のため `users` に `role`（admin/manager/staff）を持たせる。マネージャの担当範囲をチーム単位にする場合は `staff` に `manager_id`／部署列を追加。

## 5. 移行手順（概略）

1. 人事オフィスの **X-Server** 契約プランが **PHP＋MySQL 対応**か確認（X-Serverは標準で対応。要確認事項参照）。
2. MySQL スキーマ作成（上表＋users＋インデックス）。
3. 既存スプレッドシートのデータを CSV エクスポート → MySQL へ取り込み。
4. PHP で API（参照・更新・認証）を実装し、`backend.js` を fetch ベースに差し替え。
5. 認証・認可を全APIに適用。staff.html にログイン画面を追加。
6. 動作確認後、デプロイ（既存のFTPSデプロイ運用を流用可能）。

## 7. ファイル公開とシークレットの扱い

「URL直打ちでソースが見える」問題は、本番では“ファイルを隠す”のではなく、次の3層で守る設計に変わる。

- **フロント（HTML/CSS/JS）は公開前提**。秘密情報も本当の認可判定もここには置かない。見られても問題ない状態にする（ブラウザが必ずダウンロードするため、原理的に隠せない）。
- **データ保護はサーバ側APIのログイン認証で行う**。守る対象は「ファイル」ではなく「APIの応答」。JSにエンドポイントURLが見えても、有効なログインセッションが無ければデータを返さない。これが現行 JSONP 方式との最大の違い。
- **シークレット（DB接続情報・トークン等）はドキュメントルート（公開フォルダ）の外に置く**。`config.js` のようにブラウザへ配られるファイルには絶対に入れない。PHP設定ファイルはサーバ上で実行されるだけでソースは公開されず、さらに公開フォルダの外（例: `public_html` の一つ上）に置けばURLで到達できない。`.htaccess` で `deny` する方法もあるが、公開フォルダ外配置の方が確実。

「フォルダを分けて保護する」という発想は、**“Web公開してはいけないもの（設定・共通処理・取込スクリプト等）を公開フォルダの外へ出す”** という形で活きる（BASIC認証より本筋）。現リポジトリの `scripts/` はデプロイ対象から除外済みでサーバに上がっておらず、この方針と一致している。

BASIC認証の本番での位置づけは限定的で、管理画面やステージング環境の簡易ゲート程度。スタッフ・経営ダッシュボード本体は各人のログイン認証で守る。デモ期間はディレクトリ全体へのBASIC認証を「覆い」として使い、本番ではログイン認証に置き換える。

## 9. 整合性ガードとエラーメッセージ（2026-06-26 実装）

ユーザー（`jo_users`）とスタッフマスタ（`jo_staff`）は**別テーブル**で、`jo_users.staffCode → jo_staff.code`（任意・`ON DELETE SET NULL`）でリンクする。スタッフはアカウント無しでも存在でき、アカウントは紐付け無しでも存在できる（例: `hlsadmin`）。この前提のうえで、誤操作の防止と分かりやすいエラー表示を次のとおり実装した。

### 9.1 方針

- **事前チェック（pre-check）を基本**：削除・更新の前に依存件数を `SELECT COUNT` で確認し、具体的なエラーコード＋内訳を投げる。
- **二重防御**：事前チェックを潜り抜けたDB制約違反（`PDOException` SQLSTATE `23000`）は front controller の catch で捕捉し、`409 conflict / detail=constraint_violation` で返す（本番では内部詳細を出さない）。
- **コードは英語スネークケースで安定**、日本語化はフロント（`master.js` / `users.html`）の対訳辞書で実施。
- 例外設計：入力検証・運用ガード＝`InvalidArgumentException`（400 / `detail=コード`）、整合性衝突＝`JoConflictException`（409 / `detail=コード` ＋ `refs`/`otherLoginId` 等の付帯情報）。`api.php` で振り分ける。

### 9.2 マスタ code（PK）の変更不可

`staffCode`/`customerCode`/`taskCode` は工数・請求・顧客担当などに**値参照**で散在し、改番しても波及しない（＝データ破壊）。よって**編集時に code は変更不可**（`jo_upsert_master` が `oldCode != code` で `code_immutable` を投げ、UIは編集時に code 入力を `readonly`）。改番は「削除→再追加」または手動マイグレーションで対応。

### 9.3 マスタ削除の依存チェック（孤児化防止）

`jo_remove_master` は削除前に参照を集計し、1件でもあれば `master_in_use`（内訳 `refs` 付き）でブロック。運用停止は物理削除でなく **`isActive=0`** を推奨。

| マスタ | ブロックする参照（外部） | 本体と一緒に削除（子レコード） |
|---|---|---|
| staff | 工数 `jo_worklogs` / 売上目標 `jo_staff_targets` / 顧客担当 `jo_customer_staff` / ユーザー紐付け `jo_users` | — |
| customers | 工数 / 請求 `jo_billings` / 顧客担当 / 請求書 `jo_invoices` | — |
| tasks | 工数 / 請求内訳 `jo_billings.invoiceItemCode` / 請求書明細 `jo_invoice_lines` | 工程 `jo_task_phases`（トランザクションで工程→本体の順に削除） |

### 9.4 ユーザー操作のガード

- **最後の有効 admin** は削除・無効化・降格を不可（`last_admin`）。
- **自分自身**の無効化（`cannot_deactivate_self`）・管理者権限の自己剥奪（`cannot_demote_self`）・削除（`cannot_delete_self`）を不可。
- **staffCode は1ユーザーにつき一意**（`staffCode_in_use`）。アプリ事前チェック＋DB一意制約 `uq_user_staff`（NULL=紐付けなしは複数可）の二段構え。

### 9.5 エラーコード ⇔ 日本語（フロント対訳）

| コード | HTTP | 日本語 |
|---|---|---|
| `code_immutable` | 400 | コード（番号）は変更できません。 |
| `last_admin` | 400 | 最後の管理者（admin）は削除・無効化・降格できません。 |
| `cannot_deactivate_self` | 400 | ログイン中の自分自身は無効化できません。 |
| `cannot_demote_self` | 400 | ログイン中の自分自身の管理者権限は外せません。 |
| `cannot_delete_self` | 400 | 自分自身は削除できません。 |
| `staffCode_in_use` | 409 | このスタッフは既に別のユーザーに紐付いています（{otherLoginId}）。 |
| `master_in_use` | 409 | 参照があるため削除できません：{refs 内訳}。無効化での停止をご検討ください。 |
| `constraint_violation` | 409 | 関連データがあるため、この操作はできません。 |

### 9.6 関連ファイル・残課題

- 実装：`pro/lib/mutations.php`・`pro/lib/users.php`・`pro/lib/helpers.php`（`JoConflictException`）・`pro/api.php`・`pro/master.js`・`pro/users.html`。
- スキーマ：`db/schema.sql`（`uq_user_staff`）／既存DB適用は `db/migrations/2026-06-26_user_staff_unique.sql`。
- **soft-delete トグル（2026-06-26 実装）**：マスタUI（`master.html`/`master.js`）の staff/customers タブに **状態列（有効/無効）＋有効化/無効化ボタン**を追加。無効化は `upsertMaster` で `isActive=0` を送る（`{code,name,isActive}` のみ送るため顧客の住所等は upsert 対象外で保持）。tasks は `jo_task_types` に isActive 列が無いため対象外。無効行はグレー表示。
### 9.7 「無効化（isActive=0）」の定義と挙動（2026-06-26 実装・全画面）

無効化＝**“現役から外す（アーカイブ／引退）”フラグ。物理削除ではなく再有効化で復帰可能**。貫く3原則：
- **① 履歴不変**：過去の工数・請求・目標・分析・帳票・**名称表示**は無効化の影響を受けない（名称は各画面の `masterName`/`getMasterName` が全件参照で解決＝`isActive` 非依存。バックエンドJOINも同様）。
- **② 新規抑止**：今後の入力・担当割当・目標・請求の**選択肢から除外**。
- **③ 編集救済**：既存レコードが無効マスタを参照していて編集する場合、その**現在値は選択肢に残す**（`active ∪ {現在選択値}`・ラベルに「（無効）」付与）。

実装範囲（共有JS：state は全件保持＝`isActive` を捨てない／**入力セレクタのみ** active 絞り込み・**閲覧フィルタは全件のまま**）：

| 画面 | ファイル | 無効の扱い |
|---|---|---|
| 工数登録（スマホ） | `staff.js` | 顧客候補=active／作業者=自分（active）／編集中の無効顧客は補填 |
| 工数 照会・訂正 | `app.js` | 作業者・顧客とも新規=active／編集時は現在値を補填 |
| 工数登録（月間） | `worklog-month.js` | 行追加の顧客=active／スタッフpivotはactive＋現在選択 |
| 売上目標 | `targets.js` | 行=active＋当月に既存目標がある無効スタッフは表示継続 |
| 請求 照会・訂正 | `data-edit.js` | 行編集のスタッフ・顧客=active＋現在値補填。**上部の絞り込みドロップダウンは全件（履歴閲覧）** |
| 顧客担当（時系列） | `master.js` assignees | 行=有効顧客のみ／担当候補=active＋現在割当（既存割当データ・配賦は維持） |
| 経営DB・詳細分析 | `dashboard.js`/`analysis.js` | 変更なし＝全件集計（履歴・論点C） |

補足：**B**＝スタッフ無効化時、紐付く有効ログインユーザーがあれば確認ダイアログで件数・loginIdを注意表示（アカウント停止は連動せずユーザー管理で別途）。**D**＝業務区分(tasks)は `jo_task_types` に isActive 列が無く今回対象外。**E**＝無効化操作自体はブロックしない（非破壊）。
