# 本格提供に向けた認証・DB方針メモ

作成: 2026-06-18 ／ HLS（adachi）
位置づけ: デモ（静的フロント＋GAS＋スプレッドシート）から本番提供へ移行する際の、認証とデータベースの方針たたき台。実装は未着手。確定仕様ではない。
関連: [design.md](./design.md)（第7章 既知の論点／第8章 案2）

---

## 1. 背景・前提

- スタッフが工数登録するアプリ（`staff.html`）は、本番では **社員番号＋パスワードでのログイン運用**を想定。
- 本番のデータ基盤は **MySQL** を採用する想定。
- デモ段階では認証は実装しない（現行の GAS＋スプレッドシートのまま）。

MySQL（＝実バックエンド）が入る前提なら、認証は枯れた標準パターンで実装でき、難易度は低い。論点は「認証の難しさ」ではなく「GAS＋シート → 実バックエンド＋MySQL の移行をいつ・どう行うか」。

---

## 2. 想定構成

- ホスティング: 現状のお名前.com 共用レンタルサーバを継続する場合、一般に **PHP＋MySQL** は可・Node等の常駐は不可のことが多い。よって素直な構成は **バックエンド=PHP / DB=MySQL**。
- フロント: 既存の HTML/CSS/JS を**ほぼ流用**。`backend.js` の通信を JSONP から**同一オリジンの `fetch()`（PHPエンドポイント）**へ差し替える。
- 同一オリジンになるため、httpOnly・Secure な Cookie セッションが素直に使える（JSONP由来の制約が解消）。
- 将来 VPS 等に移すなら Node/Express・Python でも同設計で可。

---

## 3. 認証設計（標準パターン）

- `users` テーブル: 社員番号（ログインID）、パスワードハッシュ、権限（staff/admin）、有効フラグ、初期パスワード変更フラグ、作成/更新日時。
- パスワードは **bcrypt 等でハッシュ化**して保存（PHPなら `password_hash()` / `password_verify()`。平文保存は厳禁）。
- ログインAPIで照合 → **セッション**を発行（httpOnly・Secure・SameSite な Cookie。またはJWT）。各APIでセッション検証。
- **認可（重要）**: 画面を隠すだけでなく、各データAPIで本人・権限チェックを行う。スタッフは原則「自分の工数のみ登録・編集・削除可」。管理者は全件・マスタ操作可。
- 任意で強化: 初回ログイン時の強制パスワード変更、連続失敗時のロックアウト、操作監査ログ、パスワードリセット導線。

セキュリティ上、GAS時代の「JSONP＋URLトークン」「画面だけのガード」は本番では用いない。

---

## 4. DB移行（シート → MySQL）

現行スプレッドシートの各シートはほぼそのままテーブル化できる。

| 現シート | テーブル（案） |
|---|---|
| worklogs | worklogs |
| staff_master | staff |
| customer_master | customers |
| task_master | task_types |
| billing_data | billings |
| staff_target_master | staff_targets |
| （新規） | users（認証） |

- 集計・配賦（時間単価、達成率、工数按分、案2の主担当フォールバック）は、アプリ層のクエリ／ビューで実装。シート版のロジックを移植する。
- 工数の所有者を表すため、worklogs と users（社員番号）を紐付け、認可チェックに使う。

---

## 5. 移行手順（概略）

1. お名前.com の対象プランが **PHP＋MySQL 対応**か確認（多くは対応。要確認事項参照）。
2. MySQL スキーマ作成（上表＋users＋インデックス）。
3. 既存スプレッドシートのデータを CSV エクスポート → MySQL へ取り込み。
4. PHP で API（参照・更新・認証）を実装し、`backend.js` を fetch ベースに差し替え。
5. 認証・認可を全APIに適用。staff.html にログイン画面を追加。
6. 動作確認後、デプロイ（既存のFTPSデプロイ運用を流用可能）。

---

## 6. 要確認事項

- お名前.com の現行プランの PHP バージョン・MySQL 利用可否・DB数/容量・SSL（独自ドメインHTTPS）。
- 同時アクセス規模、バックアップ方針、個人情報（パスワード等）の取り扱い。
- GAS版と並行運用するか、一気に切り替えるか。

---

## 7. まとめ

- MySQL前提なら **社員番号＋パスワードのログインは容易**（PHPなら認証コアは数十行規模）。
- 最大の作業は認証ではなく **GAS＋シート → PHP＋MySQL の移行**。
- デモはこのまま進め、本番化の段階でスキーマ・API・認証・移行手順を本設計に落とす。
